# 🪜 실습: 자유 주제로 통과하는 PKM 사이클

> **⏰ 실습 시간**: 1회차 60분 + 2회차 90분 = 총 150분
> **🎯 실습 목표**: 자유 주제 1개를 정해 AI ↔ 인간 협업으로 *조사 → 기록 → 다듬기 → 최종 Solution*을 직접 통과한다. 1차 결과와 최종 결과를 시간 측정으로 비교해 **검증 대역폭이 품질의 상한**을 체감한다.

> **📚 학습 목표 — 이 워크숍을 마치면 다음을 할 수 있습니다:**
> - 자기 주관이 들어가는 좋은 PKM 주제를 스스로 판별할 수 있다
> - PARA + AGENTS.md 두 층 + Obsidian Skills 인프라를 설계할 수 있다
> - "AI 답이 답이 아니라 내 컨텍스트가 답"을 시간 측정으로 체감한다
> - AI 답을 그대로 쓰지 않고 **자기 컨텍스트로 재정제**하는 패턴을 익힌다
> - 평생 재사용 가능한 영구 노트 1장을 산출한다

---

## 🤔 왜 "자유 주제"인가?

같은 *"X 짜줘"* 요청도 컨텍스트가 다르면 AI가 만드는 답이 완전히 달라집니다. 하지만 **모든 주제가 워크숍에 적합한 건 아닙니다**.

워크숍은 *"AI 답을 그대로 쓰면 안 되는, 내 검증이 필요한 주제"* 일 때만 학습 효과가 나옵니다. 단순 정보 조회(*"오늘 환율"*)나 정답이 정해진 작업(*"공식 답안 있는 절차"*)은 워크숍 메시지와 안 맞습니다.

> 💡 **이 워크숍의 핵심 통찰**:
> AI 답이 답이 아니다. **내 컨텍스트가 답이다.**
> 검증할 능력이 있을 때 비로소 "검증 대역폭"이 품질의 상한이 됩니다.

## 🤔 PARA + 한국 사용자용 확장 — 8개 영역

이번 워크숍은 **PARA 방법론(Tiago Forte)** 의 4개 영역(Projects · Areas · Resources · Archives) + Inbox + 한국 대학생·취준생 워크플로우에 맞는 **확장 2개**(시스템·파일보관) = **총 8개 영역**으로 Vault를 구성합니다.

| # | 영역 | PARA 매핑 | 역할 |
|---|------|----------|------|
| 00 | **시스템** ⭐ 확장 | (PARA 외) | 메타·설정 (템플릿·AGENTS 운영 노트·Skills 가이드) |
| 01 | 수신함 | Inbox | 모든 새 정보가 처음 들어오는 임시 수집함 |
| 02 | 프로젝트 | Projects | 데드라인 있는 작업 |
| 03 | 관리영역 | Areas | 데드라인 없는 평생 관심사 |
| 04 | 자료실 | Resources | **외부 자료 원본** (PDF·웹 클리핑·인용 원문) |
| 05 | 보관소 | Archives | 완료·비활성 |
| 06 | 일일노트 | Daily | 매일 기록 |
| 07 | **파일보관** ⭐ 확장 | (PARA 외) | 마크다운 아닌 모든 파일 (이미지·PDF·코드·녹음·영상) |

> 💡 **확장 2개의 의미**:
> - **시스템(00)** = *"내 PKM을 가꾸는 메타 작업 공간"*. 템플릿·자동화·운영 노트.
> - **파일보관(07)** = *"노트가 아닌 첨부 파일"*. 노트는 다른 폴더에서 wikilink로 참조.

> 💡 **외부 자료(자료실 04) vs 정제된 노트의 위치**:
> 외부 원문(PDF·웹 클리핑)은 **자료실(04)** 에 보관하고, 그것을 읽고 *내 언어로* 정제한 영구 노트는 별도 *지식 폴더*를 두지 않고 **관리영역(03)** 의 본인 카테고리 안에 둡니다 (예: `03-관리영역/취업/나만의-자소서-원칙.md`). 평생 적용되는 *원칙·통찰*은 PARA의 *Areas* 자리이기 때문입니다.

## 🪜 폴더 만들기 — Sisyphus에게 요청

다음 프롬프트를 그대로 입력:

```
새 Obsidian Vault를 만들었어. 한국 대학생/취준생을 위한
개인 지식관리 시스템을 구축하려고 해.

PARA 방법론(Tiago Forte) + 한국 사용자용 확장에 기반해서
폴더명은 **한글로 추천**해줘
(예: "01-수신함" 같이 번호 + 한글 형식)

각 폴더 안에 README.md 파일을 만들어서 다음 3가지를 적어줘:
- **목적**: 이 폴더가 왜 존재하는가 (한 단락)
- **용도**: 무엇을 보관하는가 (구체 예시 3개 이상)
- **사용법**: 어떻게 활용하는가 (워크플로우 한 단락)

각 README는 한국어로, 마크다운으로, 본인이 한국 대학생·취준생이라
가정하고 *"내 일에 어떻게 적용할지"* 구체 예시를 포함해서.
```

### 예상 결과 (한 가지 예시)

AI가 추천한 폴더명은 학생마다 다를 수 있습니다. 예를 들면:

```
pkm-workshop/
├── 00-시스템/
│   └── README.md
├── 01-수신함/
│   └── README.md
├── 02-프로젝트/
│   └── README.md
├── 03-관리영역/
│   └── README.md
├── 04-자료실/
│   └── README.md
├── 05-보관소/
│   └── README.md
├── 06-일일노트/
│   └── README.md
└── 07-파일보관/
    └── README.md
```

> 💡 **AI가 추천한 이름이 마음에 안 들면**: 다시 요청하거나 본인이 직접 바꾸세요. *"00-System"* 같은 영문, *"00-나의시스템"* 같은 변형 모두 OK. 핵심은 *"6개월 후에도 의미가 살아있는 본인 언어"* 입니다.

Obsidian으로 돌아가서 새로 생긴 폴더가 보이는지 확인하세요. (필요시 좌측 사이드바에서 "Reload" 또는 Cmd+R)

> 💡 **PARA 메모**: 8개 README는 *AI가 만든 출발점*입니다. 워크숍이 끝난 뒤 시간 내서 본인 직업·관심사에 맞춰 다듬으세요. *"6개월 후에도 살아있는 시스템"*이 되는 핵심 작업입니다.

|---|---|---|

| `{수집함}/` | 임시 수집함 폴더 | `01-수신함/` 또는 본인이 정한 이름 |

| `{영역}/` | 지속 관리 영역 폴더 | `03-관리영역/` 또는 본인이 정한 이름 |
| `{자료}/` | 외부 자료 원본 폴더 | `04-자료실/` 또는 본인이 정한 이름 |

# Step 1. 주제 선택 (15분)

## 🎯 이 단계의 목표

- 워크숍에 적합한 *좋은 주제*가 뭔지 판별할 수 있다
- 6장 영감 카드를 보고 자기 주제를 정한다
- 맥락 차원으로 주제를 풍부하게 정의한다

> **💡 핵심 포인트**: AI가 잘 답할 수 있는 주제 ≠ 워크숍에 적합한 주제. *"내 검증이 필요한 주제"* 만 워크숍에 가치가 있습니다.

---

## 🎯 주제 선택 3가지 조건

다음 3가지 조건을 **모두** 만족하는 주제만 워크숍에 적합합니다.

### 🔍 조건 1. 조사가 필요한가?

**자가검증 질문**: *"AI 첫 답을 그대로 쓰면 위험한가?"* — YES여야 함

| ✅ 통과 | ❌ 탈락 |
|---|---|
| 1지망 회사 자소서 (인재상·직무 분석 필요) | 오늘 환율 조회 |
| 졸업논문 주제 (선행 연구·전공 동향 조사 필요) | 1+1=2 증명 |
| 첫 동남아 배낭여행 (비자·환전·치안 조사 필요) | 표준 회계 마감 절차 |

### 🤔 조건 2. 사고/판단이 필요한가?

**자가검증 질문**: *"두 사람이 같은 주제로 작업해도 결과가 다를 수 있는가?"* — YES여야 함

| ✅ 통과 | ❌ 탈락 |
|---|---|
| 대학원 진학 vs 바로 취업 (가치관 트레이드오프) | 수학 공식 풀이 |
| 동아리 환영회 기획 (참가자별 제약 조율) | 매뉴얼 따라하기 |

### 🎯 조건 3. 자기 주관이 들어가야 하는가?

**자가검증 질문**: *"이 주제를 다른 사람이 작업했다면 결과가 완전히 달라졌을까?"* — YES여야 함

| ✅ 통과 | ❌ 탈락 |
|---|---|
| 본인 분야 콘텐츠 시리즈 (취향·관점 결정적) | 누구나 동일한 객관 보고서 |
| 첫 소개팅 데이트 코스 (둘만의 컨텍스트) | 표준 양식 작성 |

> 💡 **3개 모두 YES → 좋은 주제** / 1-2개만 YES → 워크숍 효과 떨어짐 / 0개 → AI 시대 검증 대역폭 학습에 부적합

## 🎯 자가진단 — *"이 답을 그대로 쓰면 어떻게 될까?"*

받은 답을 위에서 아래로 읽으며 **4가지 질문**을 적용합니다.

### ① 시나리오 적합성 — 정말 내 시나리오에 맞나?

```
☐ AI가 가정한 컨텍스트가 내 맥락 차원과 일치하는가?
☐ 내가 명시 안 한 부분을 AI가 어떻게 채웠나? 그게 맞나?
```

**자소서 예시**: AI가 *"신입 백엔드 지원자"* 라고 가정했는데, 본인은 *"인턴 경험 1회 + 사이드 프로젝트 3개"* 였다면 — AI 답은 본인 강점을 누락한 일반론.

### ② 동선·구조 효율 — 흐름이 자연스러운가?

```
☐ 항목들의 순서가 효율적인가?
☐ 비효율적 흐름이나 중복이 있나?
```

**배낭여행 예시**: AI가 *"베트남 → 캄보디아 → 베트남"* 동선을 짰다면 비효율.

### ③ 누락 — 내가 아는 좋은 것이 빠졌나?

```
☐ 내가 아는 정보·자원·경험이 빠졌나?
☐ AI가 모를 만한 최근 정보가 빠졌나?
```

### ④ AI 한계 — AI가 모를 부분이 있나?

```
☐ 최신성 (2025년 이후 정보 부족 가능)
☐ 지역성 (한국 내수 정보 약할 수 있음)
```

### "이거 그대로 못 쓰겠다" 6가지 이유

|---|---------|------|
| 1 | ① 적합성 | AI가 *"기술 블로그를 통해"* 라고 썼지만 본인은 안 읽음 — 거짓말 |
| 2 | ① 적합성 | *"분산 시스템에 관심"* 일반론, 본인 사이드 프로젝트 3개 누락 |
| 3 | ② 구조 | 1문단 = 학력, 2문단 = 회사 칭찬, 3문단 = 포부. 본인 강점이 끼어들 자리 없음 |
| 4 | ③ 누락 | 인턴 경험 1회 + 오픈소스 컨트리뷰션 1건 — AI는 모름 |
| 5 | ④ AI 한계 | *"○○○이 최근 발표한 아키텍처"* 라고 했는데 출처 불명 — 환각 가능성 |
| 6 | ④ AI 한계 | 글자수 제한 (보통 700자) 미준수 — AI가 자유 분량으로 작성 |

> 💡 **이 발견이 핵심**: AI 답을 그대로 제출했다면 *"거짓말 + 일반론 + 본인 강점 누락 + 환각"* 4중 위험. 검증 없이 쓰는 게 얼마나 위험한지 한눈에 보입니다.

## 🤔 왜 AGENTS.md를 *두 층*으로 만드는가?

AI 에이전트는 **위치 기반으로 AGENTS.md를 자동 발견**합니다. 작업 폴더에서 시작해서 **상위로 올라가며 모든 AGENTS.md를 누적**합니다.

```
~/Documents/Vaults/pkm-workshop/

        ├── references/
        └── output/
```

> **💡 핵심 포인트**: 흩어진 조사 자료를 *"이 주제에 대한 내 생각"* 으로 응축하는 단계. 마크다운은 *"AI가 작성하기에도 인간이 읽기에도 좋은"* 공용어.

## 🪜 4b-2. 위키링크로 노트 연결

각 서브 노트도 `notes/` 안에 만들고 위키링크로 연결합니다.

```bash
# 자소서 작성자 예시
notes/
├── 01-main.md
├── 02-side-project-A.md
├── 02-side-project-B.md
├── 02-intern-experience.md
├── 02-opensource-pr.md
└── 02-failure-side-project.md
```

각 서브 노트는 *한 노트 = 한 아이디어* 원칙으로 짧게.

### Aha Moment — Graph View 확인

Obsidian 좌측 상단 Graph View 아이콘 클릭 (또는 Cmd+G):

```
[01-main] ← 모든 서브 노트가 여기로 연결됨
   ├── [02-side-project-A]
   ├── [02-side-project-B]
   ├── [02-intern-experience]
   └── [references/01-company-recruitment]
```

> 💡 **이게 PKM의 시각화**: 한 노트가 다른 5개 노트로 갈 수 있게 되면 *"내 PKM이 살아있다"* 가 그래프에 보입니다.

# Step 4c. 다듬기 — 자기 언어로 재정제 (15분)

## 🎯 이 단계의 목표

- AI가 만든 노트들을 *그대로 받지 않고* 본인 언어로 다듬는다
- 5가지 재질문 패턴으로 어색한 부분을 즉시 수정한다
- AI 답을 *재료* 로 받아 *내 자산* 으로 만드는 패턴을 익힌다

> **💡 핵심 포인트**: 이 단계가 워크숍의 *진짜 핵심*. 4a·4b의 자료가 *"AI가 만든 것"* 이라면, 4c는 *"내가 다듬은 것"* 으로 변환하는 작업.

## 🪜 4c-1. 4가지 자가진단 (5분)

`notes/01-main.md` 와 서브 노트들을 위에서 아래로 읽으며 4가지 적용:

```
☐ ① 사실 검증 — 인용한 정보가 references/ 와 일치하나?
   환각·거짓 표현 없나?

☐ ② 본인성 — 이 글이 *나만* 쓸 수 있는 글인가?
   다른 사람이 같은 회사에 지원해도 똑같이 쓸 수 있는 일반론은 X

☐ ③ 정량성 — 추상 표현을 수치로 바꿀 수 있나?
   "기여했다" → "DAU 1만 → 5만"
   "노력했다" → "주 20시간씩 3개월"

☐ ④ 인재상 매칭 — AGENTS.md의 "반드시" 항목이 다 반영됐나?
   인재상 4가지 매칭 명시?
   글자수 제한 준수?
```

## 🪜 4c-2. 5가지 재질문 패턴 (10분)

발견한 어색한 부분에 즉시 다음 패턴으로 재질문:

| 패턴 | 예시 프롬프트 (자소서 기준) |
|------|--------------------------|
| 🔍 **"왜?"** | `왜 1문단을 학력으로 시작했어? 본인 강점부터 시작하면 어떨까?` |
| 🔄 **"다른 옵션?"** | `이 표현 너무 일반론이야. 더 본인다운 표현 3개 제안해줘` |
| 🆕 **"최근?"** | `이 회사 최근 1년 안에 발표한 백엔드 관련 기술 블로그 알려줘 (정확한 출처 표기)` |
| ❌ **"이건 빼"** | `"기술 블로그를 통해" 부분 빼. 본인이 안 읽었음. 거짓말 X` |
| ➕ **"이건 추가"** | `본인 사이드 프로젝트 [[02-side-project-A]] 의 정량 지표를 1문단에 자연스럽게 녹여줘` |

### 📚 참고 예시 — 자소서 1문항 다듬기 흐름

#### Before (4b까지의 결과)

```markdown
[지원동기]

저는 ○○○ 회사의 백엔드 개발자 직무에 지원합니다.
대학에서 컴퓨터공학을 전공하며 분산 시스템과 데이터베이스에
큰 관심을 갖게 되었고, 졸업프로젝트로 마이크로서비스 기반의
주문 처리 시스템을 구현해본 경험이 있습니다.

회사의 기술 블로그를 통해 ...
```

#### 재질문 1 — "왜?"

```
왜 1문단이 학력으로 시작해? 본인 강점·경험부터 시작하는 게
인재상 "혁신" 매칭에 더 좋지 않을까?
```

#### 재질문 2 — "이건 빼"

```
"기술 블로그를 통해" 부분 빼.
본인이 안 읽었음. 거짓말 금지 (AGENTS.md 4번 규칙).
```

#### 재질문 3 — "이건 추가"

```
[[notes/02-side-project-A]] 의 "DAU 1만 → 5만 (5배 성장)" 수치를
1문단에 자연스럽게 녹여줘. 인재상 "혁신" 매칭으로.
```

#### After (4c 완료)

```markdown
[지원동기]

대학 4학년 사이드 프로젝트에서 [[02-side-project-A | 직접 만든
서비스]]가 6개월 만에 DAU 1만에서 5만으로 성장하는 경험을 했습니다.
이 과정에서 분산 캐싱과 비동기 처리로 응답 속도를 800ms에서
200ms로 단축했고, 이를 통해 *"기술이 사용자 가치로 이어지는"*
순간을 직접 경험했습니다.

이 경험은 ○○○ 회사가 추구하는 [[혁신]] 인재상 — 새로운 기술
접근으로 사용자 가치를 만드는 — 과 정확히 맞닿아 있습니다.
...
```

> 💡 **차이**: Before는 *"누구나 쓸 법한 글"*, After는 *"이 사람만 쓸 수 있는 글"*. 이게 *내 자산* 의 정의.

## 🪜 4c-3. 자기 언어로 마무리

AI가 다듬어준 결과도 한 번 더 본인 손으로 *어색한 표현·말투* 를 수정합니다.

| 작업 | 예시 |
|------|------|
| AI 어투 → 본인 어투 | "사료됩니다" → "생각합니다" |
| 너무 격식 → 본인 톤 | "심도 있게" → "깊이" |
| 영어 직역 흔적 제거 | "에 있어서" → 그냥 한국어 자연스럽게 |
| 문장 호흡 조정 | 너무 긴 문장은 두 개로 |

```bash
# {영역} 폴더(지속 관리 영역) 안에 본인 카테고리 폴더 만들기
```

> 💡 **왜 *영역* 폴더인가?** — 이번 프로젝트(*진행중* 폴더)는 끝나면 *보관소*로 이동되지만, **원칙은 평생 적용**되는 *영역*(PARA의 *Area*)에 둡니다. 이게 PARA의 진짜 가치.

### 📚 참고 예시 — 자소서 작성자의 영구 노트

```markdown
# 나만의 자소서 작성 원칙

## 발견된 원칙들

1. **AI 답을 그대로 쓰지 않는다**
   - 이유: 회사가 안 한 발표가 인용됨 (환각)
   - 적용: 모든 인용·근거를 references/ 와 대조

2. **본인 강점은 정량 지표로**
   - 이유: "기여했다" 보다 "DAU 1만→5만 (5배)"가 검증 가능
   - 적용: 모든 경험을 수치 1개 이상으로 표현

3. **회사 인재상 매칭은 1문항 1개씩 명시**
   - 이유: 4문항에 4개 매칭 = 인재상 풀 커버
   - 적용: 작성 전 인재상-경험 매트릭스 먼저
```

# 🎯 Final Aha — 워크숍의 클라이맥스

> **"AI는 95초, 나는 47분. 그 격차가 바로 품질의 상한이다."**

47분 동안 검증·다듬기를 하면서 *"AI 답을 그대로 못 쓰는 이유"* 가 끝없이 발견되는 경험. **이게 워크숍 전체의 핵심을 통과한 순간**입니다.

이 깨달음은 다음 행동을 바꿉니다:

- 🚫 AI 답을 무비판으로 복붙하는 습관 종료
- ✅ 검증 시간을 미리 예산으로 잡는 습관 시작
- ✅ AI에 더 많이 만들게 하기보다 **더 좋은 컨텍스트(AGENTS.md) + 더 좋은 도구(Skills)** 주기에 집중
- ✅ 영구 노트 1장이 다음 프로젝트에도 쓰일 가치를 알게 됨

> 💡 **5가지 패러다임 전환을 모두 통과**:
> 1. ✅ 노트 = 재료 (영구 노트 추출)
> 2. ✅ 검증 대역폭 = 품질 상한 (Before/After 측정)
> 3. ✅ PARA × Zettelkasten 하이브리드 (Project + Areas)
> 4. ✅ 마크다운 = AI 시대 공용어 (모든 작업이 .md)
> 5. ✅ AI 에이전트 + Skills 인프라 (Sisyphus + obsidian-skills 협업)
